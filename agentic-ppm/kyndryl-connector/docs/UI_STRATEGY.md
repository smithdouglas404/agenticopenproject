# UI strategy: OpenProject as datastore, Kyndral as the system of governance

Scope: Kyndral-365 DOSv2 UI (~59 pages, built on a Palantir-style ontology layer:
`/api/palantir/ontology/*`). This answers #7: how to handle the UI when some data
maps from OpenProject and other data must come from Kyndral's own setup screens.

## The clean mental model

- **OpenProject = the System of Work** (source of truth for projects, work
  packages, status, assignees, dates, time, hierarchy).
- **Kyndral = the System of Governance** on top (OKRs, divisions/segments,
  policies, custom fields, agent rules, KPIs) + the agentic intelligence.
- The bridge: OpenProject data flows **into the Palantir ontology objects**
  (`Project`, `Risk`, `Feature`, `Story`, `Task`) that the UI already reads via
  `/api/palantir/ontology/*`. So the connector's job is *not* to feed new screens
  — it's to **populate the ontology objects the existing UI already renders.**

## Three data origins — and the rule for each

| Origin | What | Where it's edited | UI behavior |
|---|---|---|---|
| **OpenProject-mapped** | projects, features, stories, tasks, status, priority, assignee, dates, time, dependencies | **OpenProject** (system of work) | Kyndral shows it (read-mostly); edits write back via the connector (bidirectional). "Open in OpenProject" deep-links. |
| **Kyndral-setup** | divisions/segments, company profile, OKRs/KPIs, custom fields, policies, agent rules, integrations | **Kyndral setup screens** | Full CRUD in Kyndral; OpenProject never supplies these. |
| **Computed/agent** | risk scores, EVM, interventions, recommendations, value realization | derived | Generated by agents; surfaced read-only with evidence + HITL approve/reject. |

## Which screens map from OpenProject vs need Kyndral setup

**Maps from OpenProject** (populate via the connector → ontology objects):
- `ProjectDetailPage` (the nexus: WBS, status, burndown, budget actuals, risks)
- `ProjectIngestionPage` (OpenProject becomes an ingestion source)
- Work/issue/dependency views, reporting/analytics over work data

**Must be Kyndral setup screens** (OpenProject can't provide these):
- **CompanyProfile** → **Business Divisions, Segments, Org Units** (the data you
  specifically called out), company metrics, strategic objectives
- **OKRManagement** → OKRs/KPIs at company/project/functional levels
- **CustomFieldManagement** → custom fields/attributes (can *map to* OpenProject
  custom fields, but defined here)
- **PolicyAsCode** → governance policies/rules
- **IntegrationManagement** → connector config (incl. the OpenProject adapter)
- **Agent config suite** (AgentManagement / DynamicAgentAdmin / AgentRules / MCP)

**Computed/agent** (no setup, no OpenProject — just surface + HITL):
- `AgentCommandCenterPage`, `AgentLens`, `ClarityChat`, `DeepAgentMonitoring`,
  `OrchestrationMonitoringPage`, `GraphExplorer`, `HITLApprovalCenter`.

## Polish plan (what to actually do)

1. **Wire OpenProject into the ontology layer, not new screens.** Implement the
   connector so OpenProject projects/WPs land as Palantir `Project`/`Feature`/
   `Story`/`Task` objects. The existing 59 pages then "just work" on real data —
   minimal UI change. (See `SCHEMA_AND_OPENPROJECT_MAPPING.md`.)
2. **Make the OpenProject-mapped fields visibly read-from-source.** On
   `ProjectDetailPage`, badge work-item fields as "from OpenProject" with an
   "Open in OpenProject" link; edits route back through the connector
   (bidirectional). This sets the right expectation: OpenProject is the datastore.
3. **Keep setup screens authoritative for governance data.** Divisions, segments,
   OKRs, policies, custom fields stay full-CRUD in Kyndral. Where they *relate* to
   work (e.g. OKR↔Epic), the link is created/inferred in Kyndral, not OpenProject.
4. **Onboarding for "no PPM tool" users (your #3 goal).** `SetupWizard` →
   CompanyProfile (divisions/segments/OKRs) → then either connect OpenProject
   *or* let them create projects directly in OpenProject (the embedded datastore).
   Either way the agents have a populated graph to reason over on day one.
5. **One HITL surface.** Route all agent actions (from any agent) through
   `HITLApprovalCenter`/`ApprovalCenter` so approve/reject is consistent, and
   mirror decisions back to OpenProject (status/comments) for teams who live there.

## The "no PPM tool" angle (why OpenProject as datastore is smart)
Teams with no PPM tool get a real one (OpenProject) for free, while Kyndral
provides the governance + agentic layer on top. The setup screens (divisions,
segments, OKRs, policies) give those teams enterprise structure they'd otherwise
lack — and because OpenProject is the datastore, everything they enter is
immediately in the ontology the agents reason over. That's the learning flywheel
you want: real users, real work, agents observing and recommending.
